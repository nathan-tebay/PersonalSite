PersonalSite

This site — a personal portfolio and documentation platform built from scratch with vanilla HTML, CSS, and JavaScript

View on GitHub

Overview

For this project I decided to utilize Claude Code as much as possible in designing, implementing, and documenting my personal website. I had several requirements in mind:

  • Use no frameworks, libraries, or build tools — just plain HTML, CSS, and JavaScript
  • Host the site on AWS Lambda using a container that could be hosted for free within the AWS Free Tier limits
  • Implement a simple CMS for blog posts and links as lightweight as possible.
  • Ensure the site was performant and responsive, with a good user experience on both desktop and mobile devices.
  • Utilize AI as much as possible in the development and documentation process.

This approach allowed me get familiar with Claude Code while creating a highly optimized personal website. I would not have implemented this with CGI as I haven't used it before. However, I very happy with the result as it is clean, fast, and meets my needs perfectly. I'm not sure Claude Code saved a lot of time, but it allowed to take a different approach and produced a less bloated codebase than using a framework would have. It also eased some of the interactions with AWS, largely taking me out of the console and end up the an automated deployment script. Overall, this was a fun and rewarding project that I'm happy with, and it was a great way to learn more about web development and AI-assisted coding. I also learned a lot about the limitations and gotchas of AWS Lambda, which was a valuable experience in itself.

Stack

Challenges

  • Lambda's filesystem is read-only except for /tmp. The web root is copied to /tmp/www at startup so httpd can serve dynamically written blog posts and manifests without a separate writable volume. Spent too much time trying to use symlinks and bind mounts to make the web root writable as Claude suggested.
  • Found out the MinIO client I was testing with does not support AWS STS temporary credentials — the --session-token flag simply does not exist. Every S3 write silently failed in Lambda making debugging difficult. Replaced with the AWS CLI, which handles the full Lambda credential chain natively however it increase the overall container size by ~100 MB.
  • At 128 MB, Lambda allocates 1/16th of a vCPU. Python startup for each AWS CLI invocation consumed the entire 30-second timeout. Increasing to 512 MB (1/4 vCPU) reduced startup to under 2 seconds per call. This is a great example of how insufficient memory can masquerade as a completely different class of bug.
  • Running the initial S3 sync synchronously in the entrypoint blocked httpd from starting, causing Lambda Web Adapter readiness timeouts. Moving the sync before exec while raising AWS_LWA_READINESS_CHECK_TIMEOUT solved both availability and ordering.
  • Using grep on a manifest preserves the leading comma from non-first entries. Passing that directly to manifest_upsert produced double commas and invalid JSON. Fixed with a sed 's/^,*//' strip before use. It took a while to get Claude on the right track with this since it kept trying to use more complex JSON parsing tools that aren't available in BusyBox.

Learnings

  • Lambda container images are a natural fit for CGI-style workloads — the process-per-request model maps directly onto Lambda invocations. This architecture is ideal for a small personal website with low to moderate traffic, and allows for a simple, stateless design. However the cold start caused by the S3 sync on startup is a significant drawback that required significant churn with Claude Code to solve.
  • Small videos will run locally but fail to play from the container due to the 6 MB Lambda payload limit. This required moving them to S3 and referencing them via CloudFront URLs, which added complexity to the deployment process.
  • Working with Claude Code has been effective for frontend development and CGI script implementation, creating and maintaining Dockerfiles and deployment scripts, and interacting with AWS to create profiles and debug issues. It failed when using Busybox httpd; it got confused with Apache's full version. I had to remind it that it had to serve CGI scripts from /var/www/http/cgi-bin; it kept trying to use the /admin/cgi-bin path and getting confused. Setting up simple authentication, it used the wrong file structure, requiring a lot of coaching to get it correct. Also, it would often leave remnants behind when refactoring and get confused when it found them later. Often Claude would stick with its original implementation and spin debugging until I gave it a different approach, which it was usually able to implement successfully. This was a good example of how AI can be a powerful tool for development, but also has limitations and requires careful guidance and oversight to ensure it produces correct and efficient code.
  • AWS temporary credentials (STS) are a hidden incompatibility surface for third-party S3 clients. The AWS CLI is the only safe choice inside Lambda. I intitial was testing with MinIO client locally, which worked fine with my long-term credentials, but failed silently in Lambda due to the use of temporary credentials. This was a major gotcha that cost a lot of time to debug.
  • Lambda CPU is strictly proportional to allocated memory. Inadequate memory manifests as slow Python startup, not OOM errors — making it look like a network or timeout bug.
  • Bundling sensible placeholder content in the container image is a cheap way to avoid blank-page states during cold starts without adding client-side retry logic. Since this is hosted on Lambda I couldn't store blog content or manifests in the image, but I was able to include placeholder manifests that are copied to the writable /tmp at startup. This way even if the S3 sync hasn't completed yet, the site can still serve content and avoid blank states.

Key Features

  • Dynamic nav and header injected via a shared layout script requiring no build step allowed for consistent updates across pages with minimal effort.
  • Multi-theme colour picker with WCAG-derived text contrast, persisted to localStorage. This was initially implemented for my development, but I decided to make it available to users as well since it was a fun feature and added a nice touch of personalization.
  • Blog system with draft/publish workflow, WIP placeholder cards and per-post image uploads. These are controlled via a simple admin panel that writes blog content and metadata to S3, which is then served by the site. This allows me to easily add new posts and update existing ones without needing to redeploy the site.
  • Categorised links page, admin-editable via the same CMS as the blog. This allows me to maintain a curated list of resources and links that I find interesting, and easily update them as needed. The links are organized into categories for easy navigation, and the admin panel allows me to quickly add, edit, or remove links as my interests evolve.
  • Cookie session for authentication to the admin panel with blog editor and links editor tabs. This provides a simple way to secure the admin functionality of the site without needing a full user management system. The session is stored in a cookie and checked on each request to the admin panel, allowing me to control access to the blog and links editing features.
  • All dynamic content backed by POSIX shell CGI scripts to my S3 bucket to keep costs low and avoid larger runtime dependencies. The AWS CLI and Lambda Web Adapter are the dependencies installed in the container image -- no Node, no PHP, no runtime.
  • S3-compatible storage with atomic local cache mirror for zero-latency reads to allow for this to scale should it become necessary as it goes viral.
  • Full S3 sync on cold start ensures content is available from the first request without needing to wait for individual fetches to populate the cache.
